1. Introduction

1.1 Purpose

The purpose of this is to describe the software requirements for the project "I Rate It". This document will outline the functional and non-functional requirements, system features, and constraints of the software. It serves as a guide for developers and testers to ensure that the final product meets the intended goals and user needs.

1.2 Document Conventions

This document follows the template for software requirements documentation according to the IEEE Software Requirements Specification

1.3 Intended Audience and Reading Suggestions

  • The intended audience for this document includes software developers involved in the "I Rate It" project. It is recommended that readers familiarize themselves with the overall project goals and objectives before diving into specific sections of this document.
  • For users, it is suggested to focus on the sections detailing user interfaces and system features to understand how the software will meet their needs.
  • For software developers, it is recommended to read the sections on functional requirements, system features, and external interfaces to grasp the technical specifications and implementation details.

    • 1.4 Product Scope

    The software aims to streamline the process of collecting, aggregating, and analyzing user ratings and reviews for various items, such as movies, books, or products. Its primary objectives are to enhance user engagement, provide reliable and transparent feedback, and support informed decision-making for both users and businesses. By offering a centralized platform for ratings and reviews, the system aligns with corporate goals of improving customer satisfaction, fostering community interaction, and leveraging user-generated data for business insights.

    1.5 Project Development Process

    The project was developed collaboratively using GitHub for version control, with each team member working on feature branches to ensure smooth integration and code review. The development started with the implementation of the Categories page, followed by the Entities page, and then the Reviews and Q&A sections. Administrative features, such as user and content management, were added in later stages. This incremental approach allowed for continuous testing and refinement of each module before moving on to the next, ensuring a stable and well-structured application.

    2. Overall Description

    2.1 Product Perspective

    The "I Rate It" project is a software product designed to provide a centralized platform for collecting, aggregating, and analyzing user ratings. It is an original solution developed to address the need for transparent and reliable feedback in digital marketplaces and communities. The diagrams below illustrate the database structure and the main components of the system (level 1 of the C4 diagram).

    2.2 Product Functions

    • Allow users to register, log in, and manage their profiles.
    • Enable users to search for and browse items.
    • Provide functionality for users to submit ratings and write reviews for items.
    • Aggregate and display overall ratings and reviews for each item.
    • Offer Questions and Answers section for each entity.
    • Offer analytics and insights based on user-generated ratings and reviews.
    • Allow administrators to manage items and users.

    2.3 User Classes and Characteristics

    • General Users:
      • Most frequent users of the system.
      • Can register, log in, search for items, submit ratings and reviews, and participate in Q&A sections.
      • Varying levels of technical expertise; interface should be intuitive and user-friendly.
      • Can access statistics.
    • Administrators:
      • Responsible for managing users and items.
      • Require advanced privileges, including access to management dashboards.
      • Higher technical expertise expected.
      • Critical for maintaining system integrity and enforcing business rules.
    • Guest Users:
      • Unregistered visitors who can browse the list of categories but cannot contribute content.
      • No special privileges; lowest security level.
      • Potential to convert to registered users.

    The most important user classes are General Users and Administrators, as they are essential for the core functionality and maintenance of the system.

    2.4 Operating Environment

    The software will operate in a local environment (localhost) without a dedicated server. It is intended to run on standard personal computers or laptops with the following characteristics:

    • Hardware Platform: Any modern PC or laptop capable of running a web browser and basic local server software (e.g., XAMPP).
    • Operating System: Compatible with Windows 10/11, macOS, and major Linux distributions.
    • Web Browser: Latest versions of Chrome, Firefox, Edge, or Safari.
    • Software Components: Requires a local web server stack such as XAMPP (Apache, PHP, MySQL) for running and testing the application.

    2.5 Design and Implementation Constraints

    • No specific regulatory or corporate policies apply.
    • Security and performance requirements are minimal due to the local, non-production environment.

    2.6 User Documentation

    This document is the only user documentation provided for the "I Rate It" system.

    2.7 Assumptions and Dependencies

    • It is assumed that users will have access to a modern web browser (Chrome, Firefox, Edge, or Safari) and a compatible operating system (Windows, macOS, or Linux).
    • The application will be deployed and operated in a local environment using XAMPP or a similar local server stack (Apache, PHP, MySQL).
    • No third-party or commercial components or APIs are used. The only external library used is Handlebars.js (via jsDelivr CDN) . This helps to render dynamic content in the application and to protect against XSS attacks.
    • Security and performance requirements are minimal due to the non-production, local deployment context.
    • If any of these assumptions change (e.g., deployment to a public server or integration with external APIs), the requirements may need to be revised.

    3. External Interface Requirements

    The design of the "I Rate It" application was chosen to prioritize ease of use, visual clarity, and accessibility for all user types. By adopting a clean and intuitive interface, users can quickly navigate the platform, find relevant information, and interact with features without confusion.

    3.1 User Interfaces

    • Categories Page: Displays the list of categories (with links to Entities Pages for users that are logged in). Includes login/register options and a form for submitting a new category, which have to be approved by the administrator.
    • Registration/Login Pages: Allows users to create an account or log in. Includes form fields for username, email, password, and validation messages.
    • User Profile Page: Shows user information, profile picture, submitted ratings/reviews, options to edit profile and the rankings in a RSS feed.
    • Entities Page: Displays the list of entities with summary information and average ratings (with links to Review Pages for each entity). Includes a form for submitting a new entity, which have to be approved by the administrator.
    • Reviews Page: Displays user reviews, and a form for submitting new ratings or reviews. Includes a Q&A section for user questions and answers.
    • Admin Dashboard: Accessible to administrators, provides statistics on user reviews and ratings.
    • Administration Pages: For managing users, categories, entities and reviews. All offer filtering and searching capabilities, as well as export (CSV, PDF, JSON). For the categories management, the administrator can also import new categories.

    3.2 Hardware Interfaces

    • Runs on standard PC or laptop hardware.
    • Relies on user input devices (keyboard, mouse, touchscreen).

    3.3 Software Interfaces

    • HTML: Used for structuring all web pages and defining the content layout of the application.
    • CSS: Provides styling, responsive design, and visual consistency across all user interfaces.
    • JavaScript: Enables dynamic content updates, form validation, and asynchronous communication with backend services (AJAX).
    • PHP: Server-side scripting language used for processing user requests, handling form submissions, and interacting with the database.

    3.4 Communications Interfaces

    • Communication between client and server is handled via HTTP/HTTPS protocols, using standard RESTful requests.
    • AJAX (Asynchronous JavaScript and XML) is used for dynamic data exchange between the frontend and backend without reloading the page.
    • Data is typically transmitted in JSON format for interoperability and ease of parsing.

    4. System Features

    4.1 User Registration and Authentication

    • Users can register by providing a unique username, email, and password.
    • Registration includes validation for unique usernames/emails and password strength.
    • Users can log in using their registered credentials.
    • Authentication is required to access personalized features (e.g., submitting reviews, managing profiles).
    • Session management ensures users remain logged in during their visit.
    • Users can log out securely at any time.
    • Administrators have separate authentication with elevated privileges.

    This feature ensures secure access to the system, protects user data, and enables personalized experiences for registered users and administrators.

    4.2 Item Rating and Review Submission

    • Registered users can submit ratings (e.g., 1–5 stars) and write textual reviews for items, as well as add photos.
    • An user can submit multiple ratings/reviews per item.
    • Reviews are displayed publicly for each entity.
    • Administrators can remove inappropriate reviews.
    • The system calculates and displays the average rating for each item based on all user submissions.

    This feature enables users to share feedback, helps others make informed decisions, and supports transparency and community engagement.

    4.3 Adding New Categories and Entities

    • Registered users can propose new categories (e.g., genres, product types) and new entities (e.g., specific movies, books, products) through dedicated forms.
    • Submitted categories and entities require administrator approval before becoming visible to all users.
    • Administrators can approve or reject proposed categories and entities.
    • Approved categories and entities are immediately available for browsing, rating, and review submission.

    This feature ensures the platform remains dynamic and user-driven, allowing the content to grow and adapt to user interests while maintaining quality and relevance through administrative oversight.

    4.4 Administration System: Dashboard and User Management

    • Admin Dashboard: Administrators have access to a dedicated dashboard that provides an overview of system statistics.
    • User Management: Administrators can view, search, filter, and manage all registered users.
    • The system allows exporting user data in various formats (CSV, PDF, JSON) for reporting or backup purposes.
    • Administrators can monitor user activity, review submission history, and enforce community guidelines by removing users who violate policies.

    These features ensure effective oversight, maintenance, and security of the platform, enabling administrators to keep the system organized, up-to-date, and compliant with operational standards.

    4.5 RSS Feed for Rankings

    • Users can subscribe to a RSS feed displaying the latest rankings or reviews.
    • Feeds are available for top and low rated entities.

    RSS feeds enhance user engagement by providing updates outside the main application.

    5. Other Nonfunctional Requirements

    5.1 Performance Requirements

    As a single page application (SPA), the system is designed to provide fast and seamless user experiences by minimizing full page reloads. Most content updates and navigation occur dynamically, resulting in quicker load times and smoother transitions for users operating in a local environment.

    5.2 Safety Requirements

    No specific safety requirements are necessary for this project, as it is intended for local use and does not handle critical or hazardous operations.

    5.3 Security Requirements

    • Password Security: All user passwords are securely encrypted (hashed) before being stored in the database to prevent unauthorized access.
    • Input Sanitization: All user input is thoroughly sanitized and validated before being stored in the database or displayed to users, protecting against malicious data.
    • Protection Against XSS: Output is properly escaped to prevent Cross-Site Scripting (XSS) attacks, ensuring that user-supplied content cannot execute scripts in the browser.
    • SQL Injection Prevention: All database queries use prepared statements or parameterized queries to prevent SQL injection attacks.

    5.4 Software Quality Attributes

    • Usability: The system is designed to be easy to use, with a simple and intuitive interface suitable for all user types.
    • Simplicity: The project maintains a straightforward structure, making it easy to understand, maintain, and extend.
    • Reliability: Basic validation and error handling ensure the system performs consistently in a local environment.
    • Maintainability: The codebase is organized and documented to facilitate future updates or modifications.
